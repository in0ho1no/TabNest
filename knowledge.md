# ロングセッション運用知見（コスト最適化）

Task 4-2〜5-8 を 1 セッションで通した経験から得た、Agent に作業を任せるときの運用知見。
**このタスクに限らず、長時間セッション全般で適用する方針**としてまとめる。
技術的な UI テストの知見は `.claude/skills/winui-uitest/` に分離してある。

---

## 背景：このセッションで判明したこと

13 タスクを 1 本の長いセッションで完走した結果、コスト内訳は次のとおり（詳細は `COST.md`）。

- 総額の約 **71% が Fable 5 の「キャッシュ読込」$254**。
- 実測 **254M ÷ 893 呼出 ≈ 1 呼出あたり 28.5 万トークンを再読込**。

犯人は「モデルが高い」ことより「**1 本の長いセッションに文脈を貯め込み続けた**」こと。

## なぜキャッシュ読込が膨らむか

プロンプトキャッシュ下では、**毎ターン、それまで積み上がった会話プレフィックス全体を 0.1× で読み直す**。
N ターン続くと読込トークンは「文脈サイズ × ターン数」、おおよそ **ターン数の 2 乗**で増える。
→ 効く対策は2軸: **(A) 文脈サイズを小さく保つ / (B) 大きな文脈の上でターンを回さない**。

---

## 運用方針（確定事項）

### 1. タスク境界で `/clear` + メモリ持ち越し（コンパクションは使わない）
- **コンパクションは使わない**（細部喪失・要約コストがあるため）。確実な `/clear` を採用。
- タスク（または Step）完了ごとに `/clear`。状態は `MEMORY.md` + メモリファイルで持ち越す。
- 効果が最も大きく、最も手軽。各セッションの文脈ピークが下がり、2 乗効果が分割される。

### 2. 重い・隔離できる作業はサブエージェントへ（A・B両方に効く）
- サブエージェントは**新しい小さな文脈で開始**し、内部で何十ターン回してもメインの巨大文脈を再読込しない。
- 対象: 探索（SPEC/既存コード読み）、GUI 評価ループ、難所のレビュー、大量検索。
- オーケストレータ（メイン）は**痩せて**いること（委譲と統合に徹し、ターン数・文脈を最小に）。

### 3. サブエージェントの戻り値は要約、成果物はファイル
- **今回の反省**: Opus レビュー13回はサブで走らせたが、**全文（各 40〜60K）がメインに戻り再読込され続けた**。
- 正解: サブが `docs/reviews/...md` を直接書き、メインには**判定＋対応必須の指摘だけ一行ずつ**返す。
- スクショ・UIA ダンプ・ビルドログも同様に**サブ内で生き死にさせ、判定文だけ返す**（画像トークンは特に重い）。

### 4. モデルルーティング（キャッシュ読込単価 = 入力の 0.1×）
| 用途 | モデル | 入力 | キャッシュ読込 |
|---|---|---:|---:|
| 検索・機械的編集・大量探索 | Haiku / Explore | $1 | $0.10 |
| 中程度の実装 | Sonnet | $3 | $0.30 |
| レビュー・難所の推論 | Opus 4.8 | $5 | $0.50 |
| オーケストレーション・最難実装のみ | Fable 5 | $10 | $1.00 |

- Fable と Haiku は **10 倍差**。ターン数の多い探索を Haiku サブに出すと二重に効く。
- 注意: **キャッシュはモデル別**。1 本の会話の途中でモデルを切り替えると無効化されるので、
  切り替えは**サブエージェント単位**で行う。

### 5. 文脈を軽く保つ
- ビルド/テストは出力 verbosity を最小に（ログ全文が文脈に残り再読込される）。
- ファイルは offset/limit で必要範囲だけ読む。同じファイルを読み直さない。
- 規約・手順は CLAUDE.md / AGENTS.md / Skill に外出しして再導出を防ぐ（ただし常時 in-context の内容は簡潔に）。

---

## タスクごとの推奨フロー

1. メイン（痩せ型）: SPEC 該当節 + 関連メモリを読み、計画。
2. 重い探索が要れば `Explore`/Haiku に委譲。
3. 実装: 軽ければインライン、重ければ Sonnet/Fable サブ。
4. build/test: 出力最小。
5. GUI 評価: `gui-evaluator` サブに委譲（スクショをサブに閉じ込め判定文だけ返す）。
6. クロスモデルレビュー: `cross-model-reviewer`（Opus）サブが review ファイルを直接書き、要約だけ返す。
7. commit / merge: メイン。
8. タスク境界で `/clear`、状態は MEMORY.md で持ち越し。

## 用意した道具（このリポジトリ）

- `.claude/agents/cross-model-reviewer.md` — 差分を SPEC と照合してレビュー、`docs/reviews/` にファイル出力、要約だけ返す（Opus）。
- `.claude/agents/gui-evaluator.md` — WinUI アプリを起動し UIA/スクショで GUI 評価、判定文だけ返す（Sonnet）。
- `.claude/agents/build-test-runner.md` — `dotnet build/test` を実行し、合否・件数・失敗要点だけ返す（Sonnet）。冗長なログをメインに入れない。
  ※ `git`/`gh` など小出力コマンドは委譲せずインライン実行（委譲はコールドスタート分だけ損になる）。
- `.claude/skills/winui-uitest/` — WinAppDriver/UIA の手順とハマりどころ（必要時にロード）。
- `AGENTS.md`「ロングセッション運用とコスト最適化」節 — 上記方針の要約と参照。

## 他プロジェクトへの展開

汎用的な `cross-model-reviewer` / `gui-evaluator` は、他プロジェクトでも使うなら
`~/.claude/agents/` にコピーすれば全プロジェクトで利用可能（`winui-uitest` Skill は WinUI 専用なので
`~/.claude/skills/` へ）。リポジトリ内の `.claude/` はチーム共有・バージョン管理向け。
